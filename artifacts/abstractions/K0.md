# Audit: K‚āÄ (Grothendieck Group) Infrastructure

## Construction

`K‚āÄ C` is defined in `BridgelandStability/GrothendieckGroup/Defs.lean` (lines 58--70)
as the quotient:

```
FreeAbelianGroup C ‚ßł K‚āÄSubgroup C
```

where `K‚āÄSubgroup C` is the `AddSubgroup.closure` of all elements
`[T.obj‚āā] - [T.obj‚āĀ] - [T.obj‚āÉ]` for distinguished triangles `T`.

**Hypotheses on `C`:**

```lean
variable (C : Type u) [Category.{v} C] [HasZeroObject C] [HasShift C ‚Ą§]
  [Preadditive C] [‚ąÄ n : ‚Ą§, (shiftFunctor C n).Additive] [Pretriangulated C]
```

This is exactly the standard K‚āÄ of a pretriangulated category. The construction
is maximally general in the sense that it does not require `IsTriangulated C`
(only `Pretriangulated`). Several downstream lemmas (e.g. `K‚āÄ.of_shift_int`) do
add `[IsTriangulated C]` when the proof requires axiom TR4.

## Downstream consumers

K‚āÄ is deeply embedded throughout the project:

1. **Stability conditions** (`StabilityCondition/Defs.lean`): The central charge lives
   on `K‚āÄ C` (or a quotient `őõ` via a class map `v : K‚āÄ C ‚Üí+ őõ`). The structures
   `PreStabilityCondition.WithClassMap` and `StabilityCondition.WithClassMap` are
   parametric in the class map.

2. **Seminorm and topology** (`StabilityCondition/Defs.lean`): `stabSeminorm` evaluates
   `U (K‚āÄ.of C E)` for semistable `E`. The Bridgeland topology is defined in terms of
   this seminorm.

3. **Skewed stability functions** (`IntervalCategory/FiniteLength.lean`):
   `SkewedStabilityFunction` carries a field `W : K‚āÄ C ‚Üí+ ‚Ąā`. Its `Semistable`
   structure checks `ssf.W (K‚āÄ.of C E) ‚Č† 0`.

4. **Deformation theory** (~14 files in `Deformation/`): Pervasive use of `K‚āÄ.of_iso`,
   `K‚āÄ.of_postnikovTower_eq_sum`, `K‚āÄ.of_shift_one`, `K‚āÄ.of_isZero` throughout the
   Harder--Narasimhan and phase-confinement proofs.

5. **Local homeomorphism / manifold** (`NumericalStabilityManifold.lean`): The main
   theorem `existsComplexManifoldOnConnectedComponent` is parametric in `v : K‚āÄ C ‚Üí+ őõ`.
   Bridgeland Corollary 1.3 specializes to `v = numericalQuotientMap`.

## Missing API

- **No `K‚āÄ.lift` uniqueness/equiv**: The universal property is stated as a forward map
  `(f : C ‚Üí A) [IsTriangleAdditive f] ‚Üí K‚āÄ C ‚Üí+ A`, but there is no converse or
  equivalence `{f : C ‚Üí A // IsTriangleAdditive f} ‚ČÉ (K‚āÄ C ‚Üí+ A)`. Uniqueness is
  instead established ad hoc via the two-step extensionality pattern
  `QuotientAddGroup.addMonoidHom_ext` followed by `FreeAbelianGroup.lift_ext`.
  This pattern appears at least 5 times across the codebase.

- **No `K‚āÄ.ext`**: There is no dedicated extensionality lemma for morphisms out of `K‚āÄ`.
  Adding `K‚āÄ.hom_ext : (‚ąÄ X, f (K‚āÄ.of C X) = g (K‚āÄ.of C X)) ‚Üí f = g` would
  eliminate the repeated `QuotientAddGroup.addMonoidHom_ext` + `FreeAbelianGroup.lift_ext`
  boilerplate.

- **No `K‚āÄ.ind`**: There is no induction principle for elements of `K‚āÄ`. This would
  help in proofs that need to work with arbitrary elements.

## Mathematical generalization assessment

### Is this the standard K‚āÄ?

Yes. The definition follows the standard construction of `K‚āÄ` for a triangulated (or
pretriangulated) category: the free abelian group on objects modulo the relation
`[B] = [A] + [C]` for every distinguished triangle `A ‚Üí B ‚Üí C ‚Üí A‚ü¶1‚üß`. The module
documentation correctly notes that the isomorphism relation is derivable (via
contractible triangles with zero third vertex).

The `HeartK0` construction separately builds `K‚āÄ` of the heart of a t-structure using
short exact sequence relations, as appropriate for an abelian category. The equivalence
`heartK0EquivK0` formalizes the classical theorem that the inclusion of the heart
induces an isomorphism on Grothendieck groups.

### Mathlib comparison

**Mathlib's `Algebra.GrothendieckGroup`** (`Mathlib.GroupTheory.MonoidLocalization.GrothendieckGroup`)
is the Grothendieck group of a commutative *monoid* `M`, defined as `Localization (top : Submonoid M)`.
This is the "K‚āÄ of a commutative monoid" construction, not a categorical K‚āÄ.

**Mathlib does not have:**
- K‚āÄ of a triangulated category
- K‚āÄ of an exact category
- K‚āÄ of an abelian category (via short exact sequence relations)
- Any categorical K-theory construction

The search for `K‚āÄ`, `KTheory`, `K_theory`, `GrothendieckGroup` in Mathlib's
`CategoryTheory/` directory yields zero results. All Mathlib hits for `K‚āÄ` are
variable names in Haar measure theory.

### Could this use `Algebra.GrothendieckGroup`?

No. `Algebra.GrothendieckGroup` group-completes a commutative monoid `(M, +, 0)`.
The triangulated K‚āÄ quotients by distinguished-triangle relations, which are not
captured by the monoid structure on objects (objects of a category don't form a monoid
under direct sum in a way that Lean's `CommMonoid` typeclass would express, and even if
they did, the triangle relations are different from the split-exact relations that the
monoid Grothendieck group captures). The current hand-rolled construction via
`FreeAbelianGroup / AddSubgroup.closure` is the correct approach and is standard.

### Potential Mathlib contribution

The `K‚āÄ` construction and `IsTriangleAdditive` typeclass are clean and general enough
to upstream to Mathlib's `CategoryTheory.Triangulated` directory. The main ingredients
(`FreeAbelianGroup`, `QuotientAddGroup`, `Pretriangulated`) are already in Mathlib.
The `HeartK0` construction could similarly live alongside the t-structure heart API.

## Generality assessment

The current K‚āÄ is at **maximal usable generality**:

- It requires only `Pretriangulated`, not `IsTriangulated`.
- The `IsTriangleAdditive` typeclass and `K‚āÄ.lift` universal property cleanly
  separate the construction from downstream use.
- The parametrization over class maps `v : K‚āÄ C ‚Üí+ őõ` in stability conditions
  is exactly right for both the full K‚āÄ and numerical quotient specializations.

The only axis of further generalization would be to define K‚āÄ for *exact categories*
(using admissible short exact sequences), which would subsume both the triangulated and
abelian cases. However, Mathlib does not yet have a notion of exact category, so this
is not actionable.
