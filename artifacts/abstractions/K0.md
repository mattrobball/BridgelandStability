# Audit: K₀ (Grothendieck Group) Infrastructure

## Construction

`K₀ C` is defined in `BridgelandStability/GrothendieckGroup/Defs.lean` as:

```lean
def K₀ : Type _ := (trianglePresentation C).K0
```

where `trianglePresentation C` is a `K0Presentation` (from
`GrothendieckGroup/Presentation.lean`) instantiated with distinguished triangles
as the relation type. The underlying quotient is
`FreeAbelianGroup C ⧸ AddSubgroup.closure {[T.obj₂] - [T.obj₁] - [T.obj₃]}`.

The `K0Presentation` abstraction factors out the identical quotient plumbing
shared by the triangulated K₀ and the heart K₀ (`HeartK0`).

**Hypotheses on `C`:**

```lean
variable (C : Type u) [Category.{v} C] [HasZeroObject C] [HasShift C ℤ]
  [Preadditive C] [∀ n : ℤ, (shiftFunctor C n).Additive] [Pretriangulated C]
```

This is exactly the standard K₀ of a pretriangulated category. The construction
is maximally general in the sense that it does not require `IsTriangulated C`
(only `Pretriangulated`). Several downstream lemmas (e.g. `K₀.of_shift_int`) do
add `[IsTriangulated C]` when the proof requires axiom TR4.

## Downstream consumers

K₀ is deeply embedded throughout the project:

1. **Stability conditions** (`StabilityCondition/Defs.lean`): The central charge lives
   on `K₀ C` (or a quotient `Λ` via a class map `v : K₀ C →+ Λ`). The structures
   `PreStabilityCondition.WithClassMap` and `StabilityCondition.WithClassMap` are
   parametric in the class map.

2. **Seminorm and topology** (`StabilityCondition/Defs.lean`): `stabSeminorm` evaluates
   `U (K₀.of C E)` for semistable `E`. The Bridgeland topology is defined in terms of
   this seminorm.

3. **Skewed stability functions** (`IntervalCategory/FiniteLength.lean`):
   `SkewedStabilityFunction` carries a field `W : K₀ C →+ ℂ`. Its `Semistable`
   structure checks `ssf.W (K₀.of C E) ≠ 0`.

4. **Deformation theory** (~14 files in `Deformation/`): Pervasive use of `K₀.of_iso`,
   `K₀.of_postnikovTower_eq_sum`, `K₀.of_shift_one`, `K₀.of_isZero` throughout the
   Harder--Narasimhan and phase-confinement proofs.

5. **Local homeomorphism / manifold** (`NumericalStabilityManifold.lean`): The main
   theorem `existsComplexManifoldOnConnectedComponent` is parametric in `v : K₀ C →+ Λ`.
   Bridgeland Corollary 1.3 specializes to `v = numericalQuotientMap`.

## Resolved API gaps

The following were identified as missing and have since been added via the
`K0Presentation` refactoring:

- **`K₀.hom_ext`** (`@[ext 1100]`): Extensionality for morphisms out of `K₀`.
  Delegates to `K0Presentation.hom_ext`. Eliminates the repeated
  `QuotientAddGroup.addMonoidHom_ext` + `FreeAbelianGroup.lift_ext` boilerplate.

- **`K0Presentation.induction_on`** (`@[elab_as_elim]`): Induction principle for
  elements of `P.K0` (and hence `K₀ C`). Cases: `of`, `zero`, `neg`, `add`.

## Remaining API gap

- **No `K₀.lift` uniqueness/equiv**: The universal property is stated as a forward map
  `(f : C → A) [IsTriangleAdditive f] → K₀ C →+ A`, but there is no converse or
  equivalence `{f : C → A // IsTriangleAdditive f} ≃ (K₀ C →+ A)`. In practice
  `K₀.hom_ext` suffices for all current use sites.

## Mathematical generalization assessment

### Is this the standard K₀?

Yes. The definition follows the standard construction of `K₀` for a triangulated (or
pretriangulated) category: the free abelian group on objects modulo the relation
`[B] = [A] + [C]` for every distinguished triangle `A → B → C → A⟦1⟧`. The module
documentation correctly notes that the isomorphism relation is derivable (via
contractible triangles with zero third vertex).

The `HeartK0` construction separately builds `K₀` of the heart of a t-structure using
short exact sequence relations, as appropriate for an abelian category. The equivalence
`heartK0EquivK0` formalizes the classical theorem that the inclusion of the heart
induces an isomorphism on Grothendieck groups.

### Mathlib comparison

**Mathlib's `Algebra.GrothendieckGroup`** (`Mathlib.GroupTheory.MonoidLocalization.GrothendieckGroup`)
is the Grothendieck group of a commutative *monoid* `M`, defined as `Localization (top : Submonoid M)`.
This is the "K₀ of a commutative monoid" construction, not a categorical K₀.

**Mathlib does not have:**
- K₀ of a triangulated category
- K₀ of an exact category
- K₀ of an abelian category (via short exact sequence relations)
- Any categorical K-theory construction

The search for `K₀`, `KTheory`, `K_theory`, `GrothendieckGroup` in Mathlib's
`CategoryTheory/` directory yields zero results. All Mathlib hits for `K₀` are
variable names in Haar measure theory.

### Could this use `Algebra.GrothendieckGroup`?

No. `Algebra.GrothendieckGroup` group-completes a commutative monoid `(M, +, 0)`.
The triangulated K₀ quotients by distinguished-triangle relations, which are not
captured by the monoid structure on objects (objects of a category don't form a monoid
under direct sum in a way that Lean's `CommMonoid` typeclass would express, and even if
they did, the triangle relations are different from the split-exact relations that the
monoid Grothendieck group captures). The current hand-rolled construction via
`FreeAbelianGroup / AddSubgroup.closure` is the correct approach and is standard.

### Potential Mathlib contribution

The `K₀` construction and `IsTriangleAdditive` typeclass are clean and general enough
to upstream to Mathlib's `CategoryTheory.Triangulated` directory. The main ingredients
(`FreeAbelianGroup`, `QuotientAddGroup`, `Pretriangulated`) are already in Mathlib.
The `HeartK0` construction could similarly live alongside the t-structure heart API.

## Generality assessment

The current K₀ is at **maximal usable generality**:

- It requires only `Pretriangulated`, not `IsTriangulated`.
- The `IsTriangleAdditive` typeclass and `K₀.lift` universal property cleanly
  separate the construction from downstream use.
- The parametrization over class maps `v : K₀ C →+ Λ` in stability conditions
  is exactly right for both the full K₀ and numerical quotient specializations.

The only axis of further generalization would be to define K₀ for *exact categories*
(using admissible short exact sequences), which would subsume both the triangulated and
abelian cases. However, Mathlib does not yet have a notion of exact category, so this
is not actionable.
