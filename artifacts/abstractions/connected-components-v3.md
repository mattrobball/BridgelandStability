# Connected Components of Stability Conditions: Architecture Design (v3)

## PHASE 1: Mathematical Content

### The objects

A **Bridgeland stability condition** on a triangulated category D consists of a pair
(Z, P) where P is a locally-finite slicing (a collection of full additive subcategories
P(phi), one per real number phi, satisfying shift-compatibility and the HN-filtration
existence axiom) and Z : K_0(D) -> C is the *central charge*, an additive homomorphism
from the Grothendieck group to the complex numbers, satisfying the compatibility
condition: for every nonzero P(phi)-semistable object E, we have Z([E]) = m * exp(i pi phi)
for some m > 0.

A **class-map stability condition** relative to a group homomorphism v : K_0(D) -> Lambda
is the same data but with the central charge factored: Z_Lambda : Lambda -> C is the
"class-lattice charge" and the ambient charge is Z = Z_Lambda . v. The data carried is
(slicing, Z_Lambda, compat). The ambient charge Z is recovered by composition.

**StabilityCondition D** is the specialization to v = id : K_0(D) -> K_0(D).
**NumericalStabilityCondition k D** specializes to v = quotient map K_0(D) -> N(D),
where N(D) = K_0(D) / rad(chi) is the numerical Grothendieck group.

### The topology

The **Bridgeland topology** on Stab(D) is generated by basis neighborhoods B_eps(sigma) =
{ tau : |Z(tau) - Z(sigma)|_sigma < sin(pi eps), d(P_sigma, P_tau) < eps }, where
|U|_sigma = sup { |U(E)| / |Z(sigma)(E)| : E nonzero sigma-semistable } is the Bridgeland
seminorm and d is the generalized slicing metric.

The topology on Stab_v(D) (class-map stability conditions relative to v) is the topology
**induced** from Stab(D) via the forgetful map sigma |-> sigma.toStabilityCondition.

### V(Sigma): the seminorm subgroup

For a connected component Sigma of Stab(D), define

  V(Sigma) = { U : K_0(D) -> C | |U|_sigma < infinity }

for any sigma in Sigma. By Lemma 6.2 (seminorm equivalence), the set V(Sigma) is
independent of the choice of sigma in Sigma, and any two such seminorms are equivalent.
V(Sigma) is a complex vector subspace of Hom_Z(K_0(D), C), and the Bridgeland seminorm
restricts to a genuine norm on V(Sigma), making it a normed complex vector space.

### Theorem 1.2 (Bridgeland)

For each connected component Sigma of Stab(D), the central charge map

  Z : Sigma -> V(Sigma),   sigma |-> Z(sigma)

is a local homeomorphism.

### Corollary 1.3

When Lambda is finitely generated and v is surjective, each connected component of
Stab_v(D) is a finite-dimensional complex manifold. The local model is the subspace of
V(Sigma) consisting of charges that factor through v; this is finite-dimensional because
Hom_Z(Lambda, C) is.

### Natural parameters

The constructions decompose into these layers:

1. **Bridgeland seminorm and slicing metric**: defined on Stab(D), intrinsically.
   Parameters: only D.

2. **V(Sigma) and its normed space structure**: defined per connected component of
   Stab(D). Parameters: D and a connected component cc of Stab(D).

3. **Theorem 1.2 (local homeomorphism)**: operates on connected components of Stab(D).
   Parameters: D and cc.

4. **Restriction to class-map conditions**: intersects a connected component of Stab(D)
   with the subspace of charges factoring through v, and restricts the local
   homeomorphism. Parameters: D, v, and a connected component of Stab_v(D) (which sits
   inside a connected component of Stab(D) via the induced topology).

5. **Finite-dimensionality and manifold structure**: requires Lambda finitely generated.
   Parameters: D, v, surjectivity of v, finite generation of Lambda.

Key observation: layers 1-3 know nothing about v. Layer 4 introduces v. Layer 5 adds
surjectivity and finite generation. This is the natural stratification.

## PHASE 2: Clean Architecture Design

### Type hierarchy

```
PreStabilityCondition.WithClassMap C v
  |
  v  (adds local finiteness)
  |
StabilityCondition.WithClassMap C v
  |                      |
  | (v = id)             | (v = numericalQuotientMap k C)
  v                      v
StabilityCondition C     NumericalStabilityCondition k C
```

Topology:
```
StabilityCondition C            -- Bridgeland topology (GenerateFrom)
  ^
  | (induced via toStabilityCondition)
  |
StabilityCondition.WithClassMap C v    -- induced topology
```

Connected component hierarchy:
```
ConnectedComponents (StabilityCondition C)     -- "ambient" components

ConnectedComponents (WithClassMap C v)          -- "class-map" components
    (maps to ambient components via toStabilityCondition)

For a fixed ambient cc:
  V(cc) := componentSeminormSubgroup C cc       -- normed C-vector space
  chargeMap : {sigma in cc} -> V(cc)            -- local homeomorphism (Thm 1.2)

For a fixed class-map cc' (mapping to ambient cc):
  The restriction of chargeMap to cc' lands in V(cc) cap {factors through v}
  This restricted map is a local homeomorphism (Cor 1.3 topological content)
  The codomain is finite-dimensional when Lambda is f.g. (Cor 1.3 algebraic content)
```

### Key definition signatures

**Layer 1: Seminorm and metric (on Stab(D))**

```lean
-- Already correct in the codebase
def slicingDist (s1 s2 : Slicing C) : ENNReal
def stabSeminorm (sigma : StabilityCondition C) (U : K0 C ->+ C) : ENNReal
def finiteSeminormSubgroup (sigma : StabilityCondition C) : AddSubgroup (K0 C ->+ C)
```

These are intrinsically at the v=id level. The seminorm formula involves
|U(K0.of C E)| / |sigma.Z(K0.of C E)| where both U and sigma.Z live in K0 C ->+ C.
There is no natural generalization to arbitrary v.

**Layer 2: V(Sigma) and norm tower (per ambient component)**

```lean
def componentSeminormSubgroup
    (cc : ConnectedComponents (StabilityCondition C)) :
    Submodule C (K0 C ->+ C)

instance componentNormedAddCommGroup
    (cc : ConnectedComponents (StabilityCondition C)) :
    NormedAddCommGroup (componentSeminormSubgroup C cc)

instance componentNormedSpace
    (cc : ConnectedComponents (StabilityCondition C)) :
    NormedSpace C (componentSeminormSubgroup C cc)
```

Parameterized by: D (via C) and an ambient connected component.
NOT parameterizable by v. The seminorm lives on K0 C ->+ C.

**Layer 3: Theorem 1.2 (per ambient component)**

```lean
-- The charge map
def componentZMap
    (cc : ConnectedComponents (StabilityCondition C)) :
    {sigma : StabilityCondition C // mk sigma = cc} ->
      componentSeminormSubgroup C cc

-- The theorem: componentZMap is a local homeomorphism
theorem componentZMap_isLocalHomeomorph
    (cc : ConnectedComponents (StabilityCondition C)) :
    IsLocalHomeomorph (componentZMap C cc)
```

This is the core of Theorem 1.2. It operates on Stab(D) only.

**Layer 4: Restriction to class-map conditions (general v)**

The class-map restriction happens in two steps:

Step 4a: Identify which charges in V(Sigma) factor through v.

```lean
def classMapFactorSubmodule (v : K0 C ->+ Lambda) :
    Submodule C (K0 C ->+ C)

-- V(Sigma) cap {factors through v}, as a submodule of V(Sigma)
def componentClassMapCodomain
    (v : K0 C ->+ Lambda)
    (cc : ConnectedComponents (StabilityCondition C)) :
    Submodule C (componentSeminormSubgroup C cc)
```

Step 4b: The class-map restriction of the local homeomorphism.

A connected component cc' of WithClassMap C v maps (via toStabilityCondition) to
an ambient connected component cc. The central charges of elements in cc' all lie
in V(cc) cap {factors through v}. The restricted charge map is a local homeomorphism
from (the image of cc' in the ambient component) to componentClassMapCodomain C v cc.

The key mathematical content: for any subspace W of V(Sigma), the restriction of a local
homeomorphism f : X -> V to f^{-1}(W) -> W is still a local homeomorphism. This is
`IsLocalHomeomorph.codRestrict_preimage`.

```lean
-- The restriction theorem
theorem classMapChargeMap_isLocalHomeomorph
    (v : K0 C ->+ Lambda)
    (cc' : ConnectedComponents (ClassMapStabilityCondition C v)) :
    IsLocalHomeomorph (classMapChargeMap C v cc')
```

Step 4c: Under surjectivity of v, WithClassMap C v is homeomorphic to
ClassMapStabilityCondition C v. So connected components correspond, and the local
homeomorphism transfers to WithClassMap C v components.

```lean
-- Surjectivity bridge
theorem WithClassMap.existsLocalHomeomorphOnComponent
    (v : K0 C ->+ Lambda) (hv : Surjective v)
    (cc : ComponentIndex C v) :
    exists (E : Type) [NormedAddCommGroup E] [NormedSpace C E]
      (f : Component C v cc -> E), IsLocalHomeomorph f
```

**Layer 5: Manifold structure (v surjective, Lambda f.g.)**

```lean
theorem WithClassMap.existsComplexManifoldOnComponent
    (v : K0 C ->+ Lambda) [AddGroup.FG Lambda] (hv : Surjective v)
    (cc : ComponentIndex C v) :
    exists (E : Type) [NormedAddCommGroup E] [NormedSpace C E]
      [FiniteDimensional C E]
      (_ : ChartedSpace E (Component C v cc)),
      IsManifold (I(C, E)) top (Component C v cc)
```

### The two-presentation question

Mathematically, Stab_v(D) can be presented as either:
- **Bundled**: WithClassMap C v, carrying Z_Lambda : Lambda ->+ C directly
- **Factorization subtype**: {sigma : StabilityCondition C // sigma.Z factors through v}

Under surjectivity of v, these are equivalent. Without surjectivity, the bundled
presentation may have multiple points mapping to the same ambient stability condition
(different Z_Lambda but same Z = Z_Lambda . v when v is not injective on Lambda).

The Theorem 1.2 proof operates on StabilityCondition C because the Bridgeland seminorm
and deformation theory are intrinsically defined there. The Corollary 1.3 proof needs to
restrict to the v-factor subspace, which is naturally a subtype of StabilityCondition C
(the factorization subtype). The connection to WithClassMap C v then requires the
surjectivity equivalence.

The clean architecture should therefore:
1. Prove Theorem 1.2 on StabilityCondition C (layers 1-3).
2. Restrict to the factorization subtype ClassMapStabilityCondition C v (layer 4).
3. Bridge to WithClassMap C v under surjectivity (layer 4c / layer 5).

This is exactly what the current code does. The bridge code is not a design flaw -- it
reflects the genuine mathematical boundary between the v-free and v-dependent parts of the
theory.

**Discrepancy 4: Theorem 1.2 Prop quantifies over `Submodule C (Lambda ->+ C)`, not `Submodule C (K0 C ->+ C)`**

`Submodule C (K0 C ->+ C)` and `sigma.Z : K0 C ->+ C`.

At v=id, Lambda = K0 C, so these coincide. For general v, the Prop CLAIMS that the
class-map charge Z : Lambda ->+ C lands in a subspace of Hom(Lambda, C) and the
restricted map is a local homeomorphism. But the PROOF only exists at v=id.

**Verdict: the general-v Prop is aspirational.** It makes a claim that has not been
independently proved for v != id. The actual content for general v goes through the
restriction machinery (NumericalStabilityManifold.lean), which works at the
StabilityCondition C level and restricts. This is fine -- the Prop correctly describes
what a proof WOULD look like at general v, even though the actual proof route goes
through v=id and restriction. No change needed.

### Bridge code analysis

The bridge chain in NumericalStabilityManifold.lean consists of:

| Bridge | Lines | Purpose | Eliminable? |
|---|---|---|---|
| `homeomorphClassMapStabilityCondition` | ~24 | WithClassMap C v ~ ClassMapStabilityCondition C v (global homeomorphism under surjectivity) | No -- genuine mathematical content (surjectivity equivalence + topology) |
| `componentIndexEquivClassMapStabilityCondition` | ~18 | Component indices correspond under the homeomorphism | No -- required by Lean's type system (components don't automatically transport) |
| `homeomorphClassMapComponent` | ~40 | Per-component homeomorphism | No -- required by Lean's type system |
| `classMapAmbientLocusHomeomorphChargePreimage` | ~40 | Identifies ClassMapAmbientLocus with charge-preimage subtype | Mixed -- some content (identifies the charge-preimage with the factor-through condition), some mechanical |
| `connectedComponentClassMapAmbientLocusRepHomeomorphClassMapComponent` | ~25 | Identifies connected component in ambient locus with ClassMapComponent | Partly mechanical (consequence of openness of ambient components + connectedness) |

Total bridge code: ~150 lines.

**Root cause**: the project maintains two presentations of class-map stability conditions
(bundled `WithClassMap` and factorization subtype `ClassMapStabilityCondition`). This is
architecturally necessary because:
- Theorem 1.2 operates at the `StabilityCondition C` level
- The factorization subtype is a subtype of `StabilityCondition C`, so restriction is
  natural
- The public API uses `WithClassMap C v`, so the final bridge is needed

Would a single-presentation architecture help? If we used only `WithClassMap C v`, we
would need to compose `toStabilityCondition` (a topological embedding under surjectivity)
with the Theorem 1.2 local homeomorphism to get a local homeomorphism on
WithClassMap C v. This would replace the ClassMapStabilityCondition-based bridge chain
with a `toStabilityCondition`-based pullback chain. The total code would be comparable.

**Verdict**: The bridge chain is not eliminable without a comparable-cost rewrite. The
code's architecture correctly reflects the mathematical structure.

### Change 3: (NOT proposed) Generalize the norm tower to general v

The norm tower (`componentSeminormSubgroup` through `componentNormedSpace`) operates on
`K0 C ->+ C` via the Bridgeland seminorm. The seminorm formula is
`sup { |U(K0.of C E)| / |sigma.Z(K0.of C E)| }` where both U and sigma.Z live in
`K0 C ->+ C`. For a class-map stability condition with Z_Lambda : Lambda ->+ C, the
ambient charge is Z_Lambda . v : K0 C ->+ C, so the seminorm still operates on
`K0 C ->+ C`.

There is no natural way to define the seminorm on `Lambda ->+ C` without composing with
v, and the resulting seminorm would not be a norm (it would have a kernel if v is not
injective on Lambda). The correct architecture is to keep the norm tower at the v=id level
and restrict the codomain for the class-map case. This is what the code does.

### Change 4: (NOT proposed) Eliminate `ClassMapStabilityCondition`

The factorization subtype `ClassMapStabilityCondition C v` = `{sigma : StabilityCondition
C // sigma.Z factors through v}` serves as the natural intermediate between
`StabilityCondition C` (where the Bridgeland theory lives) and `WithClassMap C v` (the
public-facing type). Eliminating it would require rewriting the entire
NumericalStabilityManifold.lean to work with `WithClassMap C v` directly, routing
everything through `toStabilityCondition`. The resulting code would be comparable in
length and complexity but would lose the subtype-of-StabilityCondition structure that
makes `codRestrict_preimage` applicable.

### Change 5: (NOT proposed) Merge `ComponentTopologicalLinearLocalModel` into a direct theorem

The structure `ComponentTopologicalLinearLocalModel` packages V(Sigma), its norm
instances, the charge membership proof, and the local homeomorphism into a single
non-existential record. This is consumed by NumericalStabilityManifold.lean in 6+
locations. Replacing it with a direct theorem `componentZMap_isLocalHomeomorph` would
require unpacking the same data at each consumption site. The structure is the right
abstraction.

## Summary

### Architecture assessment

The existing code's architecture is **correct** in its essential structure:

1. The type hierarchy (WithClassMap -> StabilityCondition -> NumericalStabilityCondition)
   faithfully reflects the mathematical specialization chain.

2. The norm tower operating at the StabilityCondition C level (v=id) is mathematically
   necessary -- the Bridgeland seminorm intrinsically lives on K0 C ->+ C.

3. The two-presentation design (WithClassMap + ClassMapStabilityCondition) with bridge
   code is architecturally necessary given that Theorem 1.2 operates at the v=id level
   and the manifold theorem needs to restrict to v-factoring charges.

4. The `ComponentTopologicalLinearLocalModel` structure is good API design, cleanly
   separating the local model data from its construction.

| 1 | Introduce `ConnectedComponents.rep`, eliminate `componentRep` and `classMapComponentRep` | ~65 substitutions, 2-3 files, +1 utility file | Low |
