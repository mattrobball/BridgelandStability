# Cross-Definition Analysis: StabilityCondition, IntervalCat, PostnikovTower

**Date:** 2026-03-26
**Scope:** Shared patterns across the three definitions, connections to Mathlib,
proposals for reducing maintenance burden or enabling new theorems.

## Phase 1: Critical API Summary

### StabilityCondition (StabilityCondition/Defs.lean)

```
PreStabilityCondition.WithClassMap C v
  slicing : Slicing C
  Z       : Lambda ->+ C
  compat  : forall phi E, P phi E -> ~IsZero E -> Z(v([E])) on ray exp(i*pi*phi)

StabilityCondition.WithClassMap C v
  extends PreStabilityCondition.WithClassMap C v
  locallyFinite : slicing.IsLocallyFinite C
```

Derived: `slicingDist`, `stabSeminorm`, `basisNhd`, `topologicalSpace`,
`ComponentIndex`, `Component`, `CentralChargeIsLocalHomeomorphOnConnectedComponents`.

Key facts: `WithClassMap.toStabilityCondition` forgets class map. The
abbreviation `StabilityCondition` specializes to `v = id`. The topology is
generated by `basisNhd` for epsilon in (0, 1/8).

### IntervalCat (IntervalCategory/Basic.lean)

```
Slicing.intervalProp s a b : ObjectProperty C :=
  fun E => IsZero E \/ exists (F : HNFiltration), forall i, a < F.phi i /\ F.phi i < b

Slicing.IntervalCat s a b := (s.intervalProp C a b).FullSubcategory
```

Derived: `toLeftHeart`, `toRightHeart` (fully faithful into two different
abelian hearts when `b - a <= 1`), `hasKernels`, `hasCokernels`,
`quasiAbelian`, `IsLocallyFinite`, strict Artinian/Noetherian finite length.

Key mechanism: When `b - a <= 1`, the open interval `(a,b)` embeds into
`P((a, a+1])` (left heart, controls kernels) and `P([b-1, b))` (right heart,
controls cokernels). The two-heart embedding is why the interval category is
quasi-abelian but not abelian.

### PostnikovTower (PostnikovTower/Defs.lean)

```
PostnikovTower C E
  n          : Nat
  chain      : ComposableArrows C n
  triangle   : Fin n -> Triangle C
  triangle_dist, triangle_obj1, triangle_obj2 : coherence
  base_isZero : IsZero chain.left
  top_iso    : Nonempty (chain.right ~= E)
  zero_isZero : n = 0 -> IsZero E
```

Derived: `factor i := (triangle i).obj3`, `factors`, `length`.
Extended by `HNFiltration` (adds `phi : Fin n -> R` with `StrictAnti phi` and
`semistable`). Operations: `prefix`, `ofIso`, `shiftHN`, `dropFirst`,
`dropLast`, `appendFactor`, `zero`, `single`, `phaseShift`.

## Phase 2: Duplication Analysis

### StabilityCondition vs. Mathlib's TStructure

**Searches performed:**
- leansearch: "t-structure bounded stability condition" -- returned TStructure
  basics, no stability conditions
- leanfinder: "stability function on abelian category heart of t-structure" --
  returned Abelian, TStructure, no stability functions
- leanfinder: "slicing on triangulated category Bridgeland" -- returned
  IsTriangulated, TStructure, no slicings

**Finding: No duplication.** Mathlib has `TStructure` with hearts, truncation
functors, and boundedness. It does not have:
- Slicings (R-indexed refinements of t-structures)
- Central charges
- Stability conditions
- K_0 of triangulated categories
- The Bridgeland metric or topology

The project builds a bridge: `Slicing.toTStructure` constructs a bounded
t-structure from a slicing (heart = P((0,1])). The reverse direction (bounded
t-structure + stability function on heart => slicing) is formalized in
`HeartEquivalence/`. This is Bridgeland Proposition 5.3, the fundamental
bijection between slicings and (bounded t-structure, stability function) pairs.

**Shared structure:** The project's `TStructure.HeartAbelian` proves that the
heart of a t-structure is abelian -- a result Mathlib's `Heart.lean` lists as
a TODO. This is already upstream-ready (proved in
`BridgelandStability/TStructure/HeartAbelian.lean` using
`AbelianSubcategory.abelian`).

### IntervalCat: QuasiAbelian infrastructure vs. Mathlib

**Searches performed:**
- leansearch: "quasi-abelian category strict morphism" -- returned
  `Abelian.epiWithInjectiveKernel`, `coimageImageComparison`, no quasi-abelian
- leanfinder: "coimageImageComparison is isomorphism strict morphism" --
  returned `Abelian.instIsIsoCoimageImageComparison`,
  `Abelian.coimageImageComparison`, `Abelian.ofCoimageImageComparisonIsIso`
- loogle: "QuasiAbelian" -- no results

**Finding: No duplication. ForMathlib/QuasiAbelian/Basic.lean is novel.**
Mathlib has:

| Mathlib concept | Relationship to ForMathlib |
|-----------------|---------------------------|
| `Abelian.coimageImageComparison` | Used as the base for `IsStrict` definition |
| `NormalMono` / `NormalEpi` | `IsStrictMono.normalMono` / `IsStrictEpi.normalEpi` proved |
| `RegularEpi` / `StrongEpi` | Derived from strict epi |
| `ArtinianObject` / `NoetherianObject` | Mathlib defines DCC/ACC on *all* subobjects; ForMathlib defines `IsStrictArtinianObject` / `IsStrictNoetherianObject` for DCC/ACC on *strict* subobjects specifically |

The strict/non-strict distinction matters: in a quasi-abelian (non-abelian)
category, the subobject lattice and strict-subobject lattice diverge. Finite
length in the Bridgeland sense means ACC+DCC on strict subobjects, not all
subobjects. Mathlib's `ArtinianObject`/`NoetherianObject` are not usable here.

**The `IsStrict f` definition** (`IsIso (Abelian.coimageImageComparison f)`) is
the standard notion from Schneiders (1999). Mathlib's
`Abelian.ofCoimageImageComparisonIsIso` shows that a category where *every*
morphism is strict is abelian. The ForMathlib code studies the case where only
some morphisms are strict. This is a genuine gap in Mathlib.

### PostnikovTower vs. Mathlib's SpectralObject

**Searches performed:**
- loogle: "SpectralObject" -- returned
  `CategoryTheory.Triangulated.SpectralObject C iota` with `omega_1`, `triangle`,
  `triangle_distinguished`, `precomp`, `mapTriangulatedFunctor`
- leanfinder: "SpectralObject from TStructure, filtration associated to
  t-structure" -- returned TStructure basics

**Finding: Deliberate non-duplication.** The relationship is:

| Feature | `PostnikovTower` | `SpectralObject C iota` |
|---------|------------------|------------------------|
| Indexing | `Fin n` (finite) | arbitrary category `iota` |
| Functoriality | None (object-level isos only) | Full (`omega_1 : ComposableArrows iota 1 => C`) |
| Boundary conditions | `base_isZero`, `top_iso` | None |
| Triangle coherence | Existence of isos (`Nonempty`) | Canonical (`triangle` is a function) |
| Use case | HN filtrations (finite, phase-decorated) | Spectral sequences (infinite, functorial) |

A `PostnikovTower` is *not* a special case of `SpectralObject` because it
lacks functoriality. Conversely, a `SpectralObject` restricted to `Fin (n+1)`
would carry more structure than needed. The v1 audit correctly identified this
as deliberate: the project avoids the coherence burden of functoriality because
no consumer needs it.

Mathlib uses `SpectralObject` in:
- `TStructure.SpectralObject`: the spectral object from iterated truncation
- `Abelian.SpectralObject.Basic`: spectral sequences from abelian spectral
  objects

The project imports `Mathlib.CategoryTheory.Triangulated.TStructure.SpectralObject`
in `HeartEquivalence/Basic.lean` and defines
`SpectralObject.mapHomologicalFunctor` (in `H0Homological.lean`) to push a
triangulated spectral object through a homological functor to an abelian one.
This is the connection point.

## Phase 3: Mathematical Identity Analysis

### StabilityCondition: Is the API organized around the right primitive?

**Question:** Bridgeland proved that a stability condition is equivalent to a
bounded t-structure plus a compatible stability function on the heart. The
project formalizes this via `StabilityCondition.toHeartStabilityData` (forward)
and `HeartStabilityData.toPhasePackage` (reverse). Is slicing-first the right
primitive?

**Assessment: Yes.** The slicing is the more general primitive. A t-structure is
a single integer-indexed truncation; a slicing refines this to a real-indexed
family. The project's flow is:

```
StabilityCondition = Slicing + Z + compat + locallyFinite
                          |
                    toTStructure (bounded)
                          |
                      heart (abelian)
                          |
               Z restricted to heart = stability function
```

Going the other direction (t-structure + stability function -> slicing) requires
constructing the full R-indexed family from the integer-indexed t-structure and
the stability function's HN filtrations on the heart. This reconstruction is
more complex and would be a poor primitive.

The slicing-first organization also matches the paper: Bridgeland defines
slicings in Section 5.1 and stability conditions in 5.3, with the equivalence
to (t-structure, stability function) being Proposition 5.3, a theorem rather
than a definition.

### IntervalCat: Right level of generality?

**Question:** The quasi-abelian structure holds for `P((a,b))` when `b - a <= 1`.
Should the quasi-abelian structure live on a more general "thin subcategory"?

**Assessment: No generalization needed.** The two-heart embedding that gives
the quasi-abelian structure is intrinsically tied to the phase arithmetic:
- Left heart = `P((a, a+1])`, requiring `b <= a+1`
- Right heart = `P([b-1, b))`, requiring `a >= b-1`

Both reduce to `b - a <= 1`. There is no "thin subcategory" abstraction that
would work: the width-1 constraint comes from the shift axiom `P(phi)[1] =
P(phi+1)`, which is specific to slicings. An abstract "quasi-abelian
subcategory of a triangulated category" would need to re-derive the kernel
and cokernel existence from scratch, without the two-heart mechanism.

The interval category construction P((a,b)) as `(intervalProp).FullSubcategory`
is the natural level. Every consumer in the deformation theorem (720 occurrences
across 17 files) works with `P((t - 4*epsilon_0, t + 4*epsilon_0))` for specific
centers t, always with width `8*epsilon_0 < 1`.

### PostnikovTower: Value in connecting to SpectralObject?

**Question:** `PostnikovTower` is a finite decorated `ComposableArrows` with
distinguished triangles. `SpectralObject` is the infinite functorial version.
Is there value in connecting them?

**Assessment: Not currently, but a lightweight bridge could be valuable
long-term.** The cost-benefit:

**Costs of building a bridge `PostnikovTower -> SpectralObject C (Fin (n+1))`:**
- Must add functoriality of the chain (currently only existence of isos)
- Must add coherence between chain morphisms and triangle morphisms
- Would require changing `Nonempty (_ ~= _)` to canonical isos
- Every constructor of `HNFiltration` (prefix, dropFirst, dropLast, single,
  zero, appendFactor, shiftHN) would need to produce coherent data

**Benefits:**
- Could apply `SpectralObject.mapHomologicalFunctor` directly to HN filtrations
- Could use Mathlib's spectral sequence machinery on finite filtrations
- Would connect to `TStructure.spectralObject` (the iterated truncation)

**Current workaround:** The project already uses `SpectralObject` in
`HeartEquivalence/H0Homological.lean` for the infinite functorial case. Finite
filtrations are handled ad hoc through `K0.of_postnikovTower_eq_sum` and
`ExtensionClosure.of_postnikovTower`, which walk the chain by induction.

**Verdict: Decline.** The coherence cost is high and no current consumer needs
the bridge. If Mathlib's spectral-object API grows to provide useful lemmas
about finite spectral objects (e.g., spectral sequences of finite filtrations),
revisit this.

## Phase 4: Proposals

### Proposal 1: Upstream `heartFullSubcategoryAbelian` to Mathlib

**Status: PROPOSE**

**What:** The project's `TStructure.heartFullSubcategoryAbelian` (in
`BridgelandStability/TStructure/HeartAbelian.lean`) proves that the heart of a
t-structure on a triangulated category is abelian. Mathlib's
`Mathlib.CategoryTheory.Triangulated.TStructure.Heart` has this as a TODO.

**Mechanism:** Uses `AbelianSubcategory.abelian` with two inputs:
1. `heart_h庣`: no negative Hom spaces (`Hom(iota X, (iota Y)[n]) = 0` for n < 0)
2. `heart_admissible`: every heart morphism is admissible (cone decomposes via
   truncation)

**Dependencies:**
- `TStructure/AbelianSubcategoryImageFactorisation.lean` (also ForMathlib-ready)
- Truncation functors `truncLT`, `truncGE` and their properties (already in Mathlib)

**Maintenance reduction:** Eliminates the import chain through HeartAbelian.lean
for downstream consumers. Currently, any file needing heart abelianity pulls in
the project's proof.

**Difficulty:** Medium. The proof is self-contained (~200 lines) and uses only
Mathlib's t-structure API. Requires adapting to Mathlib's style conventions.

### Proposal 2: Upstream `QuasiAbelian` class and strict morphism theory

**Status: PROPOSE**

**What:** `ForMathlib/CategoryTheory/QuasiAbelian/Basic.lean` defines:
- `IsStrict f`: `coimageImageComparison f` is an iso
- `IsStrictMono f`, `IsStrictEpi f`: mono/epi + strict
- `QuasiAbelian`: pullbacks of strict epis are strict epis, pushouts of strict
  monos are strict monos
- `StrictShortExact`: short exact with strict morphisms
- Derived: `isStrictMono_kernel`, `isStrictEpi_cokernel`, `isStrictEpi_of_normalEpi`,
  `isStrictMono_of_normalMono`, `IsStrictEpi.isIso`, `IsStrictMono.isIso`,
  `IsStrictEpi.normalEpi`, `IsStrictEpi.regularEpi`, `IsStrictEpi.strongEpi`

**Why:** These are standard definitions from Schneiders (1999), independent of
Bridgeland. Mathlib has the abelian case (where every morphism is strict) but
not the pre-abelian case. The definitions use only `Abelian.coimageImageComparison`
(already in Mathlib) and standard limits/colimits.

**Maintenance reduction:** Currently, the project must maintain ~300 lines of
quasi-abelian foundations. Upstreaming would make this Mathlib's responsibility
and allow other projects (e.g., derived categories of quasi-abelian categories)
to use the infrastructure.

**Difficulty:** Low-medium. The code is in ForMathlib, self-contained, and
follows a well-known reference. Main work is Mathlib PR formatting and
documentation.

### Proposal 3: Register `PseudoEMetricSpace` on `Slicing C`

**Status: PROPOSE (minor)**

**What:** `slicingDist` satisfies the three axioms of a `PseudoEMetricSpace`
(reflexivity, symmetry, triangle inequality). Registering this instance would
give access to Mathlib's metric space API (balls, uniform structure,
completeness, etc.) without manual re-derivation.

**Current state:** The v1 audit noted that all three axioms are proved but the
instance is not registered. The `StabilityCondition.topologicalSpace` is
currently defined by hand via `TopologicalSpace.generateFrom`.

**Maintenance reduction:** Small but consistent. Any future proof about
continuity, convergence, or completeness of the slicing metric would benefit
from the uniform structure.

**Difficulty:** Low. Just the instance declaration.

### Proposal 4: Weaken `stabSeminorm` to `PreStabilityCondition`

**Status: DECLINE**

**What:** The definition `stabSeminorm` currently takes `StabilityCondition C`
but does not use `locallyFinite`. It could be defined on
`PreStabilityCondition C`.

**Why decline:** The v1 audit noted this. However, every consumer of
`stabSeminorm` works with `StabilityCondition`, and the weakening would require
adding coercions or explicit conversions at 465+ use sites. The savings (one
fewer hypothesis) do not justify the churn.

### Proposal 5: Connect PostnikovTower to SpectralObject

**Status: DECLINE**

**Reasoning:** See Phase 3 analysis above. The coherence cost is high
(changing `Nonempty` to canonical isos, adding naturality, updating all
constructors). No current consumer needs the bridge. The project already uses
`SpectralObject` directly where needed (H0Homological.lean).

### Proposal 6: Abstract "thin subcategory" for quasi-abelian structure

**Status: DECLINE**

**Reasoning:** See Phase 3 analysis above. The two-heart mechanism is
intrinsically tied to slicing phase arithmetic. An abstract "thin subcategory"
would need to re-derive all kernel/cokernel existence from new axioms, without
gaining any consumers. The current `P((a,b))` with `b - a <= 1` is the natural
and only instance.

### Proposal 7: Strict Artinian/Noetherian to Mathlib

**Status: PROPOSE (bundled with Proposal 2)**

**What:** `IsStrictArtinianObject`, `IsStrictNoetherianObject` (DCC/ACC on
strict subobjects) are defined in ForMathlib alongside the quasi-abelian class.
They should be upstreamed together.

**Why:** Mathlib has `ArtinianObject` and `NoetherianObject` for all subobjects.
In a non-abelian preadditive category, the strict-subobject variants are the
correct notion for finite-length theory. These are small definitions (~20 lines
each) that extend Mathlib's `Subobject.ArtinianObject` module naturally.

## Cross-definition Patterns

### Pattern A: The two-heart sandwich

The central architectural pattern unifying all three definitions:

```
StabilityCondition
  contains Slicing
    which induces TStructure (via toTStructure)
      which has an abelian heart P((0,1])

IntervalCat P((a,b)) with b-a <= 1
  embeds into LeftHeart P((a, a+1])   (= heart of phaseShift(a).toTStructure)
  embeds into RightHeart P([b-1, b))  (= heart of phaseShift(b-1).toTStructureGE)

PostnikovTower
  extended to HNFiltration (with phases)
    phases determine interval membership
      interval subcategory inherits abelian kernels/cokernels from hearts
```

This pattern is the proof-theoretic engine of the entire formalization. The
quasi-abelian structure of `P((a,b))` is not intrinsic -- it is inherited from
two different abelian hearts that together cover the needed limit/colimit
structure. This is Bridgeland's key insight (Lemma 4.3) and is faithfully
formalized.

### Pattern B: K_0 additivity along filtrations

Both `PostnikovTower` and `HNFiltration` support the decomposition
`[E] = sum_i [F_i]` in K_0 (via `K0.of_postnikovTower_eq_sum`). This is used
by:
- The central charge compatibility (sum of Z(F_i) values)
- Extension closure membership (by induction on chain length)
- The deformation theorem (tracking how K_0 classes transform under deformation)

The pattern "walk the chain by induction, using `base_isZero` and
`triangle_dist`" appears in 14+ files. This is well-supported by the current
API and does not need additional abstraction.

### Pattern C: Phase arithmetic constraining category theory

The constraints `b - a <= 1`, `eta < 1/2`, `epsilon < 1/8` all arise from
interactions between:
- The shift axiom `P(phi)[1] = P(phi+1)` (width 1)
- The two-heart embedding (needs width <= 1)
- The deformation theorem (needs 8*epsilon < 1 so width <= 1)

These are arithmetic constraints that propagate through the `Fact` typeclass
mechanism (`[Fact (a < b)]`, `[Fact (b - a <= 1)]`). This design is correct:
the constraints are imposed at the point of use (instance arguments) rather
than baked into the definition, allowing the basic `IntervalCat` and
`intervalProp` to work with arbitrary intervals.
