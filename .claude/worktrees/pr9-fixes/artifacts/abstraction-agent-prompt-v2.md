# Abstraction Agent — Prompt Template v2

## When to use

One agent per definition/structure. Goal: find the right abstraction —
not weaken hypotheses, but identify shared patterns and mathematical
identity that connect to existing infrastructure.

## Prompt

```
You are analyzing the Lean 4 definition `{NAME}` in the project at
/home/matt.linux/lean/bridgeland-extract/BridgelandStability/.

Read /home/matt.linux/lean/bridgeland-extract/AGENTS.md first.

Your goal is NOT to check whether hypotheses can be weakened. Your goal is
to find the RIGHT ABSTRACTION — the one that captures the essential algebraic
or categorical pattern, eliminates duplication, and connects to existing
infrastructure.

═══════════════════════════════════════════════════════════════════
PHASE 1: Understand the definition and its critical API
═══════════════════════════════════════════════════════════════════

Read the definition at {FILE}. Then grep for `{NAME}` across the project
to catalog:
  - Every field/projection accessed by callers
  - Every constructor/introduction pattern
  - Every lemma in the `{NAME}.` namespace
  - The downstream consumers (what theorems ultimately depend on this?)

Identify the CRITICAL API: the subset of the surface that callers actually
need. Distinguish load-bearing API from convenience wrappers.

═══════════════════════════════════════════════════════════════════
PHASE 1.5: Abstraction-level audit (MANDATORY STRUCTURED OUTPUT)
═══════════════════════════════════════════════════════════════════

This codebase was partially generated by AI. AI-generated code defaults
to local optimization — building at the easiest specialization and
patching with bridges. Assume this has happened. Your job is to find
where.

STEP A — OBSERVATION (factual only, no judgments):

For `{NAME}` and every definition that consumes or produces it, fill
out this record. Do not skip any definition. One record per definition.

  DEFINITION: [name]
  FILE:LINE: [location]
  ACCEPTS: [list every type in the signature's inputs]
  RETURNS: [the output type]
  PARAMETERIZED BY: [which type parameters appear in the signature]
  COERCIONS USED INTERNALLY: [any coercion to a different type inside
    the implementation]

STEP B — ABSTRACTION MAP (factual only, no judgments):

After completing Step A, answer these questions:

  1. Does the project define a parameterized type `T v` where some
     definitions use `T v` and others use a specialization
     (e.g., `T id`, an abbrev for `T id`, or a fixed instantiation)?
     List the parameterized type and every specialization.

  2. For each definition from Step A, which level does it target —
     the parameterized type or a specialization? Make a two-column
     table: [definition | level].

  3. Are there any definitions that accept the specialization in their
     signature but whose implementation would work unchanged (or with
     only internal coercions) if the signature accepted the
     parameterized type instead? List them.

  4. Is there bridge code — functions/proofs whose sole purpose is to
     convert between the parameterized type and the specialization at
     module boundaries? List every instance with file:line.

STEP C — COMMIT:

Copy your answers to questions 3 and 4 into a section titled
"ABSTRACTION-LEVEL FINDINGS" in your output. These are your findings.
You may not retract them in later phases.

═══════════════════════════════════════════════════════════════════
PHASE 2: Duplication lens (bottom-up)
═══════════════════════════════════════════════════════════════════

Search for constructions that share the same algebraic pattern as `{NAME}`:

A. IN THIS PROJECT: Are there other definitions with the same shape?
   Same fields, same quotient structure, same inductive cases, same
   proof pattern? Look for:
   - Parallel definitions (like K₀ and HeartK0)
   - Repeated proof blocks that construct/consume the same pattern
   - Wrapper functions that exist only to reorder or adapt the API

B. IN MATHLIB AND DEPENDENCIES: Use lean_leansearch, lean_loogle,
   lean_leanfinder to search for definitions with similar:
   - Type signatures
   - Field structures
   - Universal properties
   - Mathematical names (in English, not just Lean identifiers)

   Don't just search for the exact name. Search for the PATTERN.
   If `{NAME}` is a quotient of a free group by relations, search for
   other quotient-of-free-group constructions. If it's an inductive
   with zero/mem/ext cases, search for other closure operators.

For each match, compare the API surfaces. What's shared? What differs?

═══════════════════════════════════════════════════════════════════
PHASE 3: Mathematical identity lens (top-down)
═══════════════════════════════════════════════════════════════════

What IS this definition, mathematically?

- State it in natural language, as a mathematician would in a paper
- What is the most general mathematical context where this concept
  makes sense AND where the critical API (from Phase 1) still holds?
- Name the concept in standard mathematical terminology. Search the
  literature if needed.
- Is there a known generalization in the literature that hasn't been
  formalized?

The key question: does the critical API depend on the SPECIFIC setting
(pretriangulated, ℂ-valued, etc.) or does it follow from a more
general algebraic property?

═══════════════════════════════════════════════════════════════════
PHASE 4: Propose (or decline)
═══════════════════════════════════════════════════════════════════

FIRST: Review your ABSTRACTION-LEVEL FINDINGS from Phase 1.5 Step C.
If you listed definitions whose signatures could be generalized, or
bridge code that converts between levels, then the proposal MUST
address this. Show:
  - The generalized type signatures
  - How the implementation routes through internal coercions
  - Which bridge code becomes unnecessary

If Phases 2-3 found a useful abstraction:

1. State the abstraction precisely with CONCRETE LEAN CODE SKETCHES
   - The new definition/structure
   - How existing definitions instantiate it
   - How the critical API factors through it

2. Scope it:
   - What duplication does it eliminate? (count declarations/lines)
   - What NEW instances or theorems does it enable?
   - What does it NOT touch? (category-specific content that stays)
   - Does a third party (Mathlib, another project) have a use for it?

3. Assess cost:
   - How many files change?
   - Does it add indirection that hurts readability?
   - Kill switch: "keep internal unless a third user appears" or
     "upstream-ready" or "not worth it because..."

If Phases 1.5, 2, and 3 found NOTHING:

Say "no useful abstraction found" and explain why. This is a valid
outcome. Don't invent an abstraction just to have a proposal.

═══════════════════════════════════════════════════════════════════
SELF-CHECK (mandatory, must appear in output)
═══════════════════════════════════════════════════════════════════

Before finalizing, answer these questions in your output:

1. Did I identify any observations in Phase 1.5 that I later argued
   away or softened in Phase 4? If yes, restore them as findings.

2. Did I use any of these phrases to dismiss a finding?
   - "serves different purposes"
   - "cannot be merged"
   - "genuine mathematical content"
   - "the construction is at the correct level"
   If yes, re-examine: is the phrase backed by a specific named
   design principle, or is it a rationalization?

3. For each piece of bridge code I found: did I propose eliminating
   it, or did I justify its existence? If I justified it, what
   SPECIFIC constraint makes the bridge necessary rather than a
   symptom of wrong-level construction?

═══════════════════════════════════════════════════════════════════
OUTPUT
═══════════════════════════════════════════════════════════════════

Write findings to artifacts/abstractions/{name}.md

Structure:
- Current definition (brief)
- Critical API (the subset callers need)
- ABSTRACTION-LEVEL FINDINGS (copied from Phase 1.5 Step C)
- Duplication findings (with specific file:line references)
- Mathematical identity and generalization landscape
- Proposal (or explicit "no useful abstraction")
- If proposing: concrete Lean code, scope, cost, kill switch
- SELF-CHECK answers

Do NOT run any build commands.
```

## Grouping guidance

- Run on individual definitions for focused analysis
- Group 2-3 related definitions when cross-definition patterns are likely
  (e.g., K₀ + HeartK0, StabilityCondition + IntervalCat)
- The duplication lens works best when the agent can see multiple
  definitions simultaneously
